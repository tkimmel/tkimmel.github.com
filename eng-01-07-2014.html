Wiring our lean iOS app into the core Blinker application via our API
Engineer: Don
Goals:
offload all real functionality to the core Blinker app so that the iOS application is a lightweight client
pull "transactional intelligence" from the Blinker API so that we can give app users a premium experience
reminders when it's their turn to act
tracking where they are in the transactional flow
the automation and guidance thru the transaction that's core to Blinker's promise
"Eat our own dog food" - this process is serving to tighten up our API logic as we interact with it (refactoring this logic will be a major focus in the upcoming sprint)
Context:
We envision a seamless cross-client experience down the road: if a user starts a transaction on the web, completes some steps on thier phone, and then moves back to the web, we should track all this movement and provide a seamless experience for them. Even though this won't be immediately apparent in the Denver Demo due to time constraints, building the iOS app so that it's "lean" and offloading the core logic to our API helps strengthen both the API and the iOS application for future work.

Transactional management
Engineer: Alexi, Aaron
Goals:
intelligent transactional management on behalf of buyer and seller
Context:
The Denver Demo needs to be able to manage the buyer/seller transactional flow. This is a high complecity problem, and central to the promise of replacing a human (the dealer) with our application. A simple linear checklist approach doesn't afford the transactional flexibility that we'd want to offer (for example, you probably shouldn't *have* to test drive the car before you buy it even though it's a good idea). There are also cases where a user could conceivably go backwards in the checklist (I sent an ACH, but it didn't go thru - now I have to go back and do it again). The plethora of edge cases here are what merits this being a chunk of work unto itself.
